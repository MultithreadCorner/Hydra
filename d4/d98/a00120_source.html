6 What is it?
7 -----------
8 Hydra is an header only library designed for data analysis. The library makes use of Thrust and can deploy OpenMP
9 threads, CUDA and Xeon Phi cores. It is focused on performance and precision.
10 
11 The libray core algorithms follow as close as is possible the implementations widely available in framworks like ROOT, and libraries
12 like GSL
13 
14 Main features
15 -------------
16 
17 Currently Hydra supports:
18 
19 * Generation of phase-space Monte Carlo Samples with any number of particles in the final states.
20 * Generation of sequential decays (phase-space).
21 * Sampling of multidimensional pdfs.
22 * Multidimensional fitting using binned and unbinned data sets.
23 * Evaluation of multidimensional functions over heterogeneos data sets.
24 * Numerical integration of multidimensional functions using plain or self-adaptive (Vegas-like) Monte Carlo methods.
25 
26 Hydra also provides a bunch of custom types, containers and an increasing number of algorithms
27 to maximaze performance, avoid unecessary usage of memory and grant flexibility and protability between
28 host and device calculations and deployment scenarios.
29 
30 Just changing .cu to .cpp in any source code writen only using the Hydra cosntructs is enough
31 to compile your application for OpenMP using GCC in a machine without a NVIDIA GPU installed.
32 
33 Many other possibilities and functionaties can be achivied combining the core components, bounded only by the creativity of the users.
34 
35 Hydra and Thrust
36 ----------------
37 
38 Hydra is implemented on top of the [Thrust library](https://thrust.github.io/) and rely strongly on Thrust's containers, algorithms and backend managment systems.
39 The function evaluation algorithms implemented in Hydra uses tuples to build and process datasets with mixed different types. This allow to manage
40 performance degradation associated to memory access and at same time keep flexibility.
41 
42 The official version of Thrust supports tuples with maximum ten elements. In order to overcome this limitation, Hydra uses the
43 [unoficial version, forked from the original, by Andrew Currigan and collaborators](https://github.com/andrewcorrigan/thrust-multi-permutation-iterator).
44 This version implements variadic tuples and related classes, as well as provides some additional functionalities, which are missing in the official Thrust.
45 
46 The version of Thrust distributed with Hydra is maintained by [MultithreadCorner](https://github.com/MultithreadCorner). It is basically
47 a fork of Currigan's repository, which is merged with the latest official release, distributed with the
48 [CUDA Tookit](https://developer.nvidia.com/cuda-toolkit).
49 
50 ***Hydra does not compile against the official Thrust library.***
51

53 Supported Paralels Backends
54 ---------------------------
55 
56 Hydra uses the Thrust's "backend systems" to control how the algorithms algorithms get
57 mapped to and executed on the parallel processors available to a given application.
58 There are two basic ways to access Thrust's systems: by specifying the global "device" system associated with types like ```thrust::device_vector```,
59 or by selecting a specific container associated with a particular system, such as ```thrust::cuda::vector```.
60 These two approaches are complementary and may be used together within the same program.
61 Hydra library defines the symbols ```hydra::device``` and ```hydra::host``` that controls where some algorithms are executed and
62 Hydra's specialized containers are allocated.
63 The backends are selected in compile time using the macros ```THRUST_HOST_SYSTEM``` and ```THRUST_DEVICE_SYSTEM```.
64 The following backdends are available:
65 
66 * host: THRUST_HOST_SYSTEM_CPP, THRUST_HOST_SYSTEM_OMP, THRUST_HOST_SYSTEM_TBB
67 * device: THRUST_DEVICE_SYSTEM_CPP, THRUST_DEVICE_SYSTEM_OMP, THRUST_DEVICE_SYSTEM_TBB, THRUST_DEVICE_SYSTEM_CUDA
68 
69 For example, this will compile ```my_program.cu``` using OpenMP as host backend and CUDA as device backend:
70 
71 ```bash
72 nvcc -Xcompiler -fopenmp -DTHRUST_HOST_SYSTEM=THRUST_HOST_SYSTEM_OMP -DTHRUST_DEVICE_SYSTEM=THRUST_DEVICE_SYSTEM_CUDA my_program.cu
73 ```
74 The available "host" and "device" backends can be freely combined.
75 Two important features related to the Hydra's design and the backend configuration:
76 
77 * If CUDA backend is not used, [NVCC and the CUDA runtime](https://developer.nvidia.com/cuda-toolkit) are not necessary. The programs can be compiled with GCC directly.
78 * Programs written using only Hydra, Thrust, STL and standard c++ constructs, it means without any raw CUDA code or raw calls to the CUDA runtime API,
79 can be compiled with NVCC, to run on CUDA backends, or gcc to run on OpenMP and TBB backends. **Just change the source file extention from .cu to .cpp, or something else GCC understands.**
80 
81 ###Thrust Backends###
82 
83 Thrust supports CUDA, OpenMP and TBB enabled devices, so as well Hydra.
84 Read more about backend configuration on Thrust library [here](https://github.com/thrust/thrust/wiki/Device-Backends)
85 and [here](https://github.com/thrust/thrust/wiki/Host-Backends)

107 Installation and requirements
108 -----------------------------
109 
110 Hydra is a header only library, so no build process is necessary to install it.
111 Just place the `hydra` folder and its contents where your system can find it.
112 The library run on Linux systems and requires C++11 and the varidadic version of the
113 [Thrust library](https://github.com/MultithreadCorner/thrust-multi-permutation-iterator)
114 and [boost::format](http://www.boost.org/doc/libs/1_61_0/libs/format/doc/format.html).
115 
116 Some examples demonstrating the basic features of the library are included in the `src` folder.
117 These code samples require [ROOT](https://root.cern.ch/) and [TCLAP](http://tclap.sourceforge.net/) library.
118 CUDA based targets will require a local installation of [CUDA Tookit](https://developer.nvidia.com/cuda-toolkit)
119 with version 7.5 or higher.
120 Alternatively, projects targeting [OpenMP](http://openmp.org/wp/) backend can be compiled with either nvcc or gcc.
121 The CUDA installation is not required to build the OpemMP based targets.

123 Examples
124 --------
125 
126 Some example code samples demonstrating the basic usage of the library are stored in the src directory, in the project source tree.
127 These samples can be built using [CMAKE](https://cmake.org/) according the following instructions:
128 
129 1. clone the git repository: `git clone https://github.com/MultithreadCorner/Hydra.git`
130 2. go to Hydra directory: `cd Hydra`
131 3. create a build directory: `mkdir build`
132 4. go to build directory: `cd build`
133 4. `cmake ../`
134 5. `make`
135 
136 
137 The examples are named according to the convention `HYDRA_Example_<BACKEND AND COMPILER>_<EXAMPLE NAME>`. To run an example do `./example-name`.
138 The examples are described below:
139 
140 1. __PhaseSpace__ : Takes arguments from the command line and generates a 3-body decay and calculates some observables.
141 The program print some events and timing information to sdtout.
142 
143 2. __Evaluate__ : Takes arguments from the command line, generates some samples and perform calculations
144 using lambda functions (requires CUDA 8.0 to run on the GPU).
145 The program print some results and timing information to sdtout.
146 
147 3. __Fit__: Takes arguments from the command line, generates a samples and perform a extended likelihood fit.
148 The program print some results and timing information to sdtout.
149 
150 4. __Random__: Takes arguments from the command line, generates some samples in one, two and three.
151 The program print some results, draw plots and timing information to sdtout.
